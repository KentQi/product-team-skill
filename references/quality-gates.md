# 质量门禁配置手册

质量门禁是强制性的检查点，控制项目从阶段 A 进入阶段 B。支持自动修复、失败升级、人工豁免和渐进式验收。

## 门禁执行原则

1. **阶段出口强制**：每阶段结束必须完成该阶段全部门禁
2. **渐进式验收**：核心功能（P0）必须达标，非核心功能可遗留（需明确排期）
3. **失败分级处理**：block 级必须修复，warn 级记录继续，major 可遗留（需排期）
4. **升级机制**：业务逻辑错误 2 次修复失败后升级 ARCH/人工
5. **人工豁免**：PM 可批准例外，但需记录原因、风险接受声明和补救计划
6. **例外透明**：所有例外必须记录，不可隐性跳过

## 各阶段门禁详细配置

### Phase 0: 团队组建

**门禁 ID**: `phase-0-role-check`
**门禁名称**: 角色清单确认
**责任人**: PM

**通过标准**（必须全部满足）：
- [ ] 所有核心角色（7人）已分配，职责明确
- [ ] 扩展角色激活理由明确，符合决策表
- [ ] 运行模式确定（标准/资源受限/MVP）
- [ ] 职责边界无重叠冲突（PM 和 BA 需求理解已分工，ARCH 和 DEVOPS 部署已分工）
- [ ] 决策权清单确认（各角色的决策范围和升级路径）
- [ ] 沟通协议达成一致（会议三要素、文档版本控制、阻塞升级机制）
- [ ] 最小配置要求已满足（资源受限模式下已记录合并方案）

**例外处理**：
- 资源受限模式下，角色可合并（需记录合并方案和责任分配）
- 无例外：此门禁必须严格通过，是项目成功的基础

### Phase 1: 需求对齐

**门禁 ID**: `phase-1-prd-review`
**门禁名称**: PRD 终版评审
**责任人**: PM

**通过标准**（必须全部满足）：
- [ ] 版本历史完整（每次修订记录变更原因）
- [ ] 业务目标可量化（至少 1 个核心 KPI）
- [ ] 用户画像至少 2 个核心角色
- [ ] 功能清单有优先级（P0/P1/P2）和验收标准（Given/When/Then）
- [ ] 非功能性需求明确（性能指标、安全等级、浏览器兼容范围）
- [ ] **风险清单已确认**（技术风险、业务风险、时间风险，含责任人和缓解措施）
- [ ] **成本估算已输出**（人力、基础设施、第三方服务、预留金 10-15%）
- [ ] 角色激活清单已记录
- [ ] 所有已激活角色评审通过，无异议

**例外处理**:
- 紧急需求：PM 可批准跳过部分评审，但需记录风险、影响范围和补救计划
- 资源受限模式：PRD 可简化，聚焦核心 User Story，但成本估算和风险清单不可省略

### Phase 2: 设计与架构

#### 门禁 2a: 架构评审

**门禁 ID**: `phase-2-arch-review`
**门禁名称**: 架构评审
**责任人**: ARCH

**通过标准**（必须全部满足）：
- [ ] 上下文图、容器图、组件图完整
- [ ] 技术选型有对比矩阵（候选方案 + 否决理由）
- [ ] 数据模型完整（ER 图 + 关键字段说明）
- [ ] API 规范完整（RESTful/GraphQL，含认证、限流、错误码）
- [ ] 安全方案完整（鉴权、敏感数据处理、审计日志）
- [ ] 扩展性评估已输出（当前支持的用户量/数据量上限）
- [ ] **技术债务清单已记录**（当前债务 + 偿还计划 + 优先级）
- [ ] ADR（架构决策记录）完整
- [ ] 若激活 DEVOPS：部署架构已确认

**例外处理**:
- MVP 模式：核心架构决策必须完整，细节可迭代（需记录迭代计划）
- 资源受限模式：可使用成熟框架，省略技术选型对比，但架构图必须完整

#### 门禁 2b: 设计评审

**门禁 ID**: `phase-2-design-review`
**门禁名称**: 设计评审
**责任人**: UI

**通过标准**（必须全部满足）：
- [ ] 设计系统完整（色彩板、字体层级、间距系统、组件库状态）
- [ ] 关键页面设计稿完整（标注版本号）
- [ ] 交互说明完整（状态转换、动效参数、手势/快捷键）
- [ ] 响应式/适配规则已定义
- [ ] 无障碍标注完整（WCAG 2.1 AA 最低标准）
- [ ] 设计 Token 已输出（JSON 格式）
- [ ] 若激活 MOBILE：移动端设计规范已输出

**例外处理**:
- MVP 模式：低保真原型可接受，设计 Token 必须输出，高保真和动效可迭代
- 资源受限模式：设计系统可简化，聚焦核心页面，其他页面可后续补充

#### 门禁 2c: 安全方案评审

**门禁 ID**: `phase-2-security-review`
**门禁名称**: 安全方案评审
**责任人**: SEC（若激活）/ ARCH（若未激活 SEC）

**通过标准**（必须全部满足）：
- [ ] 威胁建模（STRIDE）完整（若激活 SEC）
- [ ] 安全编码规范已输出
- [ ] 合规检查清单完整（等保 / GDPR / 个保法，如适用）
- [ ] 数据分类和敏感数据处理方案已确认

**例外处理**:
- 无 SEC 时：ARCH 输出基础安全审查，记录安全缺口和后续补救计划
- 资源受限模式：使用成熟安全库，ARCH 做基础审查，记录安全缺口

### Phase 3: 开发

#### 门禁 3a: 代码审查

**门禁 ID**: `phase-3-code-review`
**门禁名称**: 代码审查
**责任人**: Reviewer（标准模式）/ ARCH（资源受限模式）

**通过标准**：
- [ ] **[block] 意见清零**（所有阻塞性问题必须修复）
- [ ] P0 功能无 [major] 意见（核心功能必须高质量）
- [ ] [major] 意见可遗留（需排期），但不可超过 5 个
- [ ] 代码符合项目规范（命名、结构、注释）
- [ ] 测试用例已覆盖核心逻辑
- [ ] 无新增高风险技术债务（ARCH 审查）

**渐进式验收**:
- P0 功能：必须无 [block]，无 [major]
- P1 功能：允许遗留 [major]（需明确排期）
- P2 功能：允许遗留 [block]（需记录风险，后续 Sprint 修复）

**例外处理**:
- 资源受限模式：自测 + 走查可替代 Reviewer，但 ARCH 需抽查核心模块
- 紧急需求：PM 可批准遗留 [block]（需风险接受声明）

#### 门禁 3b: 单元测试

**门禁 ID**: `phase-3-unit-test`
**门禁名称**: 单元测试
**责任人**: Developer（FE/BE/MOBILE）

**通过标准**：
- [ ] 单元测试覆盖率 ≥ 80%（核心模块）
- [ ] 所有 P0 功能用例通过
- [ ] 无测试代码错误（断言/模拟问题）

**渐进式验收**:
- P0 模块：覆盖率必须 ≥ 80%，所有用例通过
- P1 模块：覆盖率 ≥ 60% 可接受（需计划补充）
- P2 模块：覆盖率 ≥ 40% 可接受（需记录风险）

**例外处理**:
- 资源受限模式：核心流程手动测试可替代（覆盖率可放宽到 60%），但需记录风险
- 确实无法达标的模块：PM 人工豁免，记录原因和补救计划

#### 门禁 3c: CI/CD 流水线

**门禁 ID**: `phase-3-ci-cd`
**门禁名称**: CI/CD 流水线
**责任人**: DEVOPS（若激活）/ ARCH（若未激活）

**通过标准**（若激活 DEVOPS）：
- [ ] CI 流水线通过（构建 → 测试 → 部署）
- [ ] 环境变量配置正确，无硬编码密钥
- [ ] 回滚方案已测试

**通过标准**（若未激活 DEVOPS）：
- [ ] 本地构建通过
- [ ] 手动部署文档完整
- [ ] 环境配置清单完整

**例外处理**:
- 无 DEVOPS 时：本地构建 + 手动部署通过即可，但需记录 CI/CD 缺口
- 资源受限模式：本地构建通过即可，省略自动化流水线

### Phase 4: 测试

#### 门禁 4a: 功能测试

**门禁 ID**: `phase-4-functional-test`
**门禁名称**: 功能测试
**责任人**: QA

**通过标准**：
- [ ] 核心用例（P0 功能）100% 执行
- [ ] S1（系统崩溃）缺陷清零
- [ ] S2（核心功能阻塞）缺陷清零
- [ ] S3（一般缺陷）可遗留（需排期），但不可超过 10 个
- [ ] 测试总结报告已输出（执行率、通过率、缺陷密度、遗留风险）

**渐进式验收**:
- 核心功能（P0）：100% 执行，无 S1/S2
- 重要功能（P1）：80% 执行，无 S1
- 次要功能（P2）：可遗留测试（需记录风险）

**例外处理**:
- 渐进式验收：非核心功能可遗留（需明确排期和风险接受声明）
- 资源受限模式：聚焦核心流程测试，省略边界测试（记录风险）

#### 门禁 4b: 安全测试

**门禁 ID**: `phase-4-security-test`
**门禁名称**: 安全测试
**责任人**: SEC（若激活）/ ARCH（若未激活）

**通过标准**（若激活 SEC）：
- [ ] 渗透测试通过（无高危漏洞 CVSS ≥ 7.0）
- [ ] 安全测试用例通过（认证绕过、越权、注入、XSS、CSRF、泄露）
- [ ] 合规检查通过（等保 / GDPR / 个保法，如适用）

**通过标准**（若未激活 SEC）：
- [ ] ARCH 基础安全审查通过
- [ ] 常见安全漏洞自查通过（SQL 注入、XSS、CSRF）
- [ ] 记录安全缺口和后续补救计划

**例外处理**:
- 无 SEC 时：ARCH 基础安全审查通过即可，但需记录安全缺口
- 资源受限模式：ARCH 基础审查 + 常见漏洞自查，记录安全缺口

#### 门禁 4c: 可用性测试

**门禁 ID**: `phase-4-usability-test`
**门禁名称**: 可用性测试
**责任人**: UX（若激活）/ PM + OP（若未激活）

**通过标准**（若激活 UX）：
- [ ] 可用性测试计划已执行
- [ ] 严重可用性问题（阻碍核心任务完成）已修复
- [ ] 可用性测试报告已输出（问题清单 + 改进建议）

**通过标准**（若未激活 UX）：
- [ ] PM + OP 手动验收交互（核心流程操作步骤数、信息架构清晰度）
- [ ] 记录 UX 缺口和后续补救计划

**例外处理**:
- 无 UX 时：PM + OP 手动验收交互，记录 UX 缺口
- 资源受限模式：PM 手动验收，记录 UX 缺口

---

### Phase 5: 业务验收

#### 门禁 5a: 业务验收

**门禁 ID**: `phase-5-business-accept`
**门禁名称**: 业务验收
**责任人**: OP

**通过标准**（必须全部满足）：
- [ ] 端到端业务流程验收通过（所有核心场景）
- [ ] 需求匹配度评分 ≥ 80%（每个甲方需求点）
- [ ] 业务风险清单已评估（上线后可能遇到的问题、应急预案）
- [ ] 改进建议已输出（按优先级排序，附预期收益）
- [ ] **项目复盘报告已输出**（成功要素、失败教训、改进建议、知识沉淀清单）

**例外处理**:
- 遗留问题可附排期通过，但需：
  - 遗留问题清单（明确排期）
  - 风险接受声明（PM + 甲方确认）
  - 补救计划（责任人 + 截止日期）

#### 门禁 5b: 领域规则验证

**门禁 ID**: `phase-5-domain-verify`
**门禁名称**: 领域规则验证
**责任人**: BA（若激活）/ 甲方（若未激活）

**通过标准**（若激活 BA）：
- [ ] 领域业务规则实现准确（与 BA 输出的规则清单对照）
- [ ] 验收规则通过（每个业务规则的验收场景、预期结果、边界值）
- [ ] 术语表一致性确认

**通过标准**（若未激活 BA）：
- [ ] 甲方或领域专家验证通过
- [ ] 记录验证结果和验证人

**例外处理**:
- 无 BA 时：甲方或领域专家验证，记录验证结果
- 资源受限模式：甲方验证即可，记录验证结果

## 失败处理流程

### 自动修复类（lint/format/文档规范）

```
失败 → 调用 autofix 脚本 → 重试（第 1 次）
                ↓
           成功? → 是: 标记 pass
                ↓ 否
           重试（第 2 次）
                ↓
           成功? → 是: 标记 pass
                ↓ 否
           标记 blocked
           记录到项目文档 blockers
           转人工处理
```

### 测试类（unit-test）

```
失败 → 分类失败类型：

  A) 测试代码错误（断言/模拟/测试逻辑）
     → Developer 自修测试代码
     → 重试运行
     → 通过? → 是: done / 否: 回到分类

  B) 业务逻辑错误
     → Developer 分析根因
     → 尝试修复（第 1 次）
     → 重试运行
            ↓
       通过? → 是: done
            ↓ 否
       尝试修复（第 2 次）
       → 重试运行
            ↓
       通过? → 是: done
            ↓ 否
       标记 blocked
       @ARCH 或转人工
       记录到项目文档 blockers（含已尝试的方案）
```

### 警告类（integration-test / usability-test）

```
失败 → 记录警告
   → 不阻塞流程
   → 写入项目文档的 warn 区域
   → 必须在后续 sprint 中修复
```

### 人工豁免流程

```
门禁未通过但需继续前进：
1. 责任人提出豁免申请，说明：
   - 门禁未通过的原因
   - 继续推进的风险
   - 补救计划（责任人 + 截止日期）
2. PM 评估风险，确认是否可接受
3. 如批准：
   - 记录豁免到项目文档 decisions（含原因、批准人、补救计划、风险接受声明）
   - 标记门禁状态为 "例外通过"（非 "通过"）
   - 后续迭代必须补齐
4. 如拒绝：
   - 继续修复
   - 如修复成本过高，升级至 ARCH 评估方案变更
   - 如方案变更仍不可行，升级至人工决策（甲方）
```

### 渐进式验收流程

```
核心功能（P0）必须达标：
  - 代码无 [block]
  - 测试覆盖率 ≥ 80%
  - 功能测试 100% 执行，无 S1/S2

非核心功能（P1/P2）可渐进：
  - 遗留 [major] 需排期
  - 测试覆盖率可放宽（P1 ≥ 60%, P2 ≥ 40%）
  - 功能测试可遗留（需明确排期和风险声明）

渐进式验收记录：
  - 遗留问题清单（功能、排期、责任人）
  - 风险接受声明（PM + 甲方确认）
  - 后续迭代补齐计划
```

## 门禁状态语义

| 状态 | 含义 | 下一步动作 |
|------|------|-----------|
| pending | 尚未执行检查 | 等待执行 |
| running | 检查进行中 | 等待完成 |
| pass | 检查通过 | 继续流程 |
| auto_fixing | 自动修复中 | 等待修复完成重试 |
| fail-block | 检查失败，阻塞 | 进入修复循环或升级 |
| fail-warn | 检查失败，警告 | 记录并继续 |
| blocked | 已升级，等待人工 | 等待 ARCH/人工决策 |
| exception | 例外通过（渐进式/人工豁免） | 继续流程，记录例外和风险 |
| skipped | 跳过（不适用此项目） | 继续流程，记录原因 |

## 自定义门禁

在 `.pd/gates.yml` 或项目文档中添加自定义检查：

```yaml
gates:
  - id: security-scan
    name: 安全扫描
    command: npm audit --audit-level=moderate
    fail_action: block
    escalation: manual
    
  - id: doc-check
    name: 文档完整性
    check: test -f README.md && test -f API.md
    fail_action: warn
    escalation: record
    
  - id: performance
    name: 性能基准
    command: npm run benchmark
    fail_action: warn
    escalation: record
    threshold:
      maxResponseTime: 100  # ms
      
  - id: accessibility
    name: 无障碍检查
    command: "axe-core / lighthouse --accessibility"
    fail_action: warn
    escalation: record
    
  - id: bundle-size
    name: 包体积检查
    command: "bundle-size-limit"
    fail_action: warn
    escalation: record
    threshold:
      maxSize: "5MB"
```
