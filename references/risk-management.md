# 风险管理手册

系统化的风险管理流程，覆盖风险识别、评估、应对、跟踪和升级全生命周期。

## 风险识别

### 风险识别时机

| 阶段 | 风险类型 | 识别责任人 | 识别方法 |
|------|---------|-----------|---------|
| Phase 0-1 | 业务风险、需求风险、资源风险 | PM | 需求评审、利益相关者访谈、预算评估 |
| Phase 1-2 | 技术风险、架构风险、安全风险 | ARCH | 技术选型评审、威胁建模、架构评审 |
| Phase 2-3 | 进度风险、依赖风险、人员风险 | PM | 进度计划评审、依赖矩阵分析、资源评估 |
| Phase 3-4 | 质量风险、测试覆盖风险、性能风险 | QA + ARCH | 测试计划评审、覆盖率分析、性能基准 |
| Phase 4-5 | 上线风险、业务连续性风险、合规风险 | OP + SEC | 验收测试、应急预案评审、合规检查 |

### 常见风险清单（按类别）

#### 业务风险

| 风险项 | 典型表现 | 识别信号 |
|--------|---------|---------|
| 需求变更频繁 | 甲方反复修改需求 | PRD 版本号 > 5，变更原因多为"甲方要求" |
| 需求理解偏差 | 交付物与甲方预期不符 | 验收时大量"这不是我要的"反馈 |
| 竞品抢占市场 | 竞品提前发布类似功能 | 竞品动态监控显示功能重叠 |
| 业务目标不可量化 | 成功标准模糊 | PRD 中业务目标为"提升用户体验"等定性描述 |
| 利益相关者冲突 | 多方需求矛盾 | 不同部门对同一功能提出矛盾要求 |

#### 技术风险

| 风险项 | 典型表现 | 识别信号 |
|--------|---------|---------|
| 技术选型失误 | 框架/工具不满足需求 | 技术选型后 POC 失败 |
| 架构扩展性不足 | 用户增长后系统崩溃 | 架构评估显示当前上限低于预期峰值 |
| 第三方依赖风险 | 依赖库停止维护或有漏洞 | 依赖库长期未更新，或发现 CVE |
| 性能瓶颈 | 响应时间超标 | 性能测试显示 P99 > 目标值 150% |
| 数据一致性风险 | 并发场景下数据错误 | 缺乏事务保障或分布式锁 |
| 安全漏洞 | 渗透测试发现高危漏洞 | CVSS ≥ 7.0 的漏洞 |

#### 进度/资源风险

| 风险项 | 典型表现 | 识别信号 |
|--------|---------|---------|
| 关键人员离职 | 核心开发者/ARCH 离开 | 人员请假频繁，或已提离职 |
| 进度延误 | 里程碑未按时完成 | 实际进度 < 计划进度 80% |
| 依赖延迟 | 外部团队/接口未按时交付 | 依赖方进度红灯 |
| 预算超支 | 实际成本 > 估算 | 实际成本 > 估算 × 110% |
| 范围蔓延 | 功能不断增加 | Sprint 中新增未计划任务 > 20% |

### 风险识别方法

```
1. 头脑风暴：项目启动时，全体角色参与风险识别会议
2. 检查清单：使用本手册的"常见风险清单"逐项检查
3. 假设分析：列出项目假设，逐一验证假设的可靠性
4. 历史复盘：参考类似项目的历史风险（知识库）
5. 专家访谈：领域专家（BA、SEC）识别行业特有风险
6. 利益相关者访谈：甲方、用户、运营识别业务风险
```

## 风险评估

### 评估矩阵

```
          影响程度
        低(1)  中(2)  高(3)
      ┌───────┬───────┬───────┐
  高(3) │  中(3) │  高(6) │ 极高(9) │  ← 必须立即制定缓解计划，升级至 PM + ARCH
概    ├───────┼───────┼───────┤
率  中(2) │  低(2) │  中(4) │  高(6)  │  ← 需制定缓解计划，持续跟踪
      ├───────┼───────┼───────┤
  低(1) │  低(1) │  低(2) │  中(3)  │  ← 记录监控，无需主动干预
      └───────┴───────┴───────┘

风险等级 = 概率 × 影响
- 极高 (7-9)：红色，必须立即处理
- 高 (5-6)：橙色，需制定详细缓解计划
- 中 (3-4)：黄色，记录并定期监控
- 低 (1-2)：绿色，接受，无需额外投入
```

### 概率评估标准

| 等级 | 概率范围 | 定义 |
|------|---------|------|
| 高 | 60-100% | 极可能发生，已有明显迹象 |
| 中 | 30-60% | 可能发生，有一定迹象 |
| 低 | 0-30% | 不太可能发生，无明显迹象 |

### 影响评估标准

| 维度 | 高影响 | 中影响 | 低影响 |
|------|--------|--------|--------|
| 进度 | 延误 > 2 周 | 延误 1-2 周 | 延误 < 1 周 |
| 成本 | 超支 > 20% | 超支 10-20% | 超支 < 10% |
| 质量 | 核心功能不可用 | 重要功能受限 | 次要功能受影响 |
| 安全 | 数据泄露/系统瘫痪 | 安全漏洞需修复 | 安全建议 |
| 业务 | 项目取消/重大损失 | 业务目标部分未达成 | 业务目标轻微偏差 |

## 风险应对策略

| 策略 | 适用场景 | 成本 | 示例 |
|------|---------|------|------|
| **规避** | 高概率 + 高影响，且可通过方案变更避免 | 中高 | 更换不成熟技术栈，取消低价值功能，调整交付范围 |
| **转移** | 高影响，但可通过外包/保险转移 | 中 | 安全测试外包给第三方，购买云服务 SLA，购买保险 |
| **缓解** | 中高概率，可通过增加投入降低概率或影响 | 中 | 增加代码审查轮次，补充测试用例，增加备份方案，提前培训 |
| **接受** | 低概率 + 低影响，或无法规避/转移/缓解 | 低 | 记录风险，定期监控，建立应急储备（预算/时间），不投入额外资源 |

### 风险跟踪表

| 风险 ID | 描述 | 类别 | 等级 | 策略 | 责任人 | 状态 | 更新日期 |
|---------|------|------|------|------|--------|------|---------|
| risk-001 | 第三方 API 停止服务 | 技术 | 高 | 缓解 | ARCH | tracking | 2026-06-24 |
| risk-002 | 甲方需求变更频繁 | 业务 | 高 | 缓解 | PM | tracking | 2026-06-24 |
| risk-003 | 核心开发者离职 | 进度 | 中 | 缓解 | PM | open | 2026-06-24 |

### 跟踪频率

| 风险等级 | 跟踪频率 | 汇报对象 |
|---------|---------|---------|
| 极高 | 每日 | PM + ARCH + 人工决策 |
| 高 | 每周 | PM + 相关角色 |
| 中 | 每阶段 | PM |
| 低 | 每阶段 | 记录即可 |

### 风险状态定义

| 状态 | 定义 | 转换条件 |
|------|------|---------|
| open | 新识别，尚未制定应对计划 | 识别后立即标记 |
| tracking | 已制定应对计划，正在执行 | 制定应对计划后 |
| resolved | 风险已消除 | 缓解措施生效，确认风险不再存在 |
| accepted | 风险无法消除，已接受 | 决策层确认接受 |
| escalated | 已升级至更高层级处理 | 当前层级无法处理 |
| obsolete | 风险不再相关（项目变更） | 项目范围/环境变更 |

## 风险升级机制

```
层级 1：识别责任人（Developer/QA/UI 等）
  → 记录到风险清单
  → 初步评估概率和影响
  → 提出应对建议

层级 2：PM 评估
  → 确认风险等级
  → 制定缓解计划，分配资源
  → 纳入项目状态跟踪
  → 中低风险在此层级处理

层级 3：PM + ARCH 联合评估
  → 重大技术风险或资源风险
  → 评估技术方案可行性
  → 调整架构或计划
  → 高风险在此层级处理

层级 4：人工决策（甲方/管理层）
  → 极高风险，影响项目存亡
  → 涉及预算重大调整或项目取消
  → 涉及合规或法律问题
  → 最终仲裁者
```

### 升级触发条件

| 触发条件 | 升级至 | 时限 |
|---------|--------|------|
| 极高风险识别 | 层级 4（人工决策） | 立即 |
| 高风险 3 天无缓解进展 | 层级 3（PM + ARCH） | 3 天内 |
| 风险导致里程碑延误 > 1 周 | 层级 3（PM + ARCH） | 立即 |
| 技术风险影响架构可行性 | 层级 3（PM + ARCH） | 立即 |
| 安全风险涉及数据泄露 | 层级 4（人工决策） | 立即 |
| 预算超支 > 20% | 层级 4（人工决策） | 立即 |
| 中风险 2 周无缓解进展 | 层级 2（PM） | 2 周内 |

## 风险与门禁的关联

| 风险类别 | 关联门禁 | 门禁失败信号 |
|---------|---------|------------|
| 技术风险 | phase-2-arch-review | 架构评审反复失败 |
| 质量风险 | phase-3-unit-test, phase-4-functional-test | 测试覆盖率不达标，缺陷率高 |
| 进度风险 | phase-3-code-review | 代码审查反复失败，修复周期长 |
| 安全风险 | phase-2-security-review, phase-4-security-test | 渗透测试发现高危漏洞 |
| 业务风险 | phase-5-business-accept | 验收匹配度 < 80% |

## 行业特定风险（补充）

### 金融行业

- 合规风险：监管政策变化、等保/密评要求
- 数据风险：敏感金融数据泄露、交易数据一致性
- 系统风险：交易系统可用性要求（99.99%）、熔断机制

### 医疗行业

- 合规风险：医疗器械认证、HIPAA/等保
- 数据风险：患者隐私数据（PII）、医疗记录准确性
- 业务风险：临床使用安全、误诊风险

### 政务行业

- 合规风险：等保三级、数据主权、国产化要求
- 流程风险：审批流程长、需求变更受政策影响
- 集成风险：与现有政务系统集成的复杂性

### 电商/零售

- 峰值风险：大促期间流量峰值（双11、黑五）
- 支付风险：支付接口稳定性、资金安全
- 物流风险：库存同步、订单履约
