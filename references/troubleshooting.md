# 故障排除与常见问题

常见问题解答（FAQ）、典型案例、故障恢复流程和合规检查清单。

## 常见问题解答（FAQ）

### Q1: 项目启动时如何快速判断需要激活哪些扩展角色？

**A**: 使用快速决策树（见 SKILL.md 触发条件部分），10 秒内判断：
1. 是否需要多环境部署 / 高可用 → 激活 DEVOPS
2. 是否涉及复杂交互 / 多步骤流程 → 激活 UX
3. 是否处理支付 / PII 数据 / 合规 → 激活 SEC
4. 是否包含 App / 小程序 → 激活 MOBILE
5. 是否涉及算法 / 大数据 / AI → 激活 DATA
6. 领域知识是否超出 PM 理解范围（金融/医疗/政务） → 激活 BA

### Q2: 资源受限时（只有 2-3 人），如何配置角色？

**A**: 使用资源受限模式（见 references/resource-constrained.md）：
- 3 人模式：PM+OP / ARCH+BE+DEVOPS / UI+FE+QA
- 2 人模式：全栈开发者 + 设计师
- 1 人模式：全栈开发者（极简流程）

### Q3: PM 和 BA 在需求理解上重叠怎么办？

**A**: 明确分工：
- 激活 BA 后，PM 负责协调和需求变更管理，BA 负责深度领域分析
- PM 不替代 BA 做领域建模，BA 不替代 PM 做需求优先级决策
- 冲突时：PM 拥有最终解释权（业务维度），但需更新 PRD 消除歧义

### Q4: 质量门禁太严格，导致项目无法推进怎么办？

**A**: 使用渐进式验收和人工豁免：
1. 区分核心功能（P0）和非核心功能（P1/P2）
2. P0 必须达标，P1/P2 可遗留（需明确排期）
3. 如确实无法达标，PM 可批准人工豁免（记录原因、风险接受声明、补救计划）
4. 资源受限模式下，门禁标准可进一步放宽（但需记录风险）

### Q5: 技术债务如何管理？

**A**: 技术债务管理三步法：
1. **识别**：Phase 2 ARCH 输出技术债务清单（当前债务 + 偿还计划）
2. **跟踪**：每个 Sprint 分配 10-20% 工时偿还技术债务
3. **偿还**：高严重度债务优先，新代码不引入已知类型债务

### Q6: 如何处理远程团队的协作挑战？

**A**: 远程协作三原则：
1. 文档即沟通：所有状态变更和决策必须体现在文档中
2. 异步优先：使用文档、PR、设计评论进行异步协作
3. 每日同步：PM 每天更新项目状态摘要，作为团队日报
详细规范见 references/knowledge-management.md 远程协作部分

### Q7: 项目成本超支怎么办？

**A**: 成本超支处理流程：
1. PM 分析超支原因（范围蔓延？估算不准？意外风险？）
2. 提出调整方案：削减范围（降低 P1/P2 优先级）、延长时间、增加资源
3. 预警（超支 10-20%）：PM 内部调整，跟踪后续成本
4. 超支（>20%）：升级至甲方/管理层确认

### Q8: 多个角色拥有"最终决策权"产生冲突怎么办？

**A**: 使用决策升级机制：
1. 层级 1：角色间协商
2. 层级 2：PM 初步协调
3. 层级 3：ARCH 技术仲裁 / PM 业务决策 / SEC 安全否决
4. 层级 4：人工决策（甲方/管理层）最终仲裁
所有层级 3+ 决策必须记录 ADR（架构决策记录）

### Q9: 如何防止需求频繁变更？

**A**: 需求变更管理：
1. PM 是唯一需求入口，所有变更须经 PM 确认
2. 使用结构化语言："需求编号 + 需求描述 + 验收标准"
3. 变更必须记录版本历史，说明变更原因
4. 重大变更（影响成本/进度 > 10%）需全体角色重新评审
5. Sprint 进行中，原则上不接受新需求（可记录到下一 Sprint）

### Q10: 安全测试未通过，但上线时间紧迫怎么办？

**A**: 安全不可妥协，但可分级处理：
1. 高危漏洞（CVSS ≥ 7.0）：必须修复，不可豁免
2. 中危漏洞（CVSS 4.0-6.9）：制定修复计划，可先上线但需监控
3. 低危漏洞（CVSS < 4.0）：可记录到后续迭代修复
4. 所有豁免必须经过 SEC（或 ARCH）+ PM 确认，记录风险接受声明

## 典型案例

### 案例 A：成功项目（标准模式）

**项目背景**：电商小程序，标准团队 7+2 人（激活 DEVOPS + MOBILE），周期 8 周

**成功要素**：
1. Phase 0 角色激活准确：DEVOPS 和 MOBILE 激活理由明确
2. Phase 1 PRD 完整：成本估算和风险清单提前识别了支付接口风险
3. Phase 2 架构设计预留了扩展性：支持后续接入新支付渠道
4. Phase 3-4 敏捷迭代：开发→测试→修复循环高效，每日站会同步
5. Phase 5 验收深入：OP 模拟了 12 个端到端场景，发现 3 个业务逻辑问题

**关键经验**：
- 提前识别支付接口风险，准备了备用方案（支付宝 + 微信支付双通道）
- 技术债务清单记录了旧版 SDK 的兼容问题，Sprint 3 偿还
- 项目复盘沉淀了电商小程序的通用 PRD 模板

### 案例 B：失败教训（需求蔓延）

**项目背景**：企业协作工具，标准团队，周期 12 周（原计划 8 周）

**失败原因**：
1. Phase 1 PRD 不完整：非功能性需求模糊，导致后续大量补充
2. 需求变更未受控：Sprint 3 后甲方每周新增 2-3 个功能，未走变更流程
3. 成本估算未包含预留金：需求蔓延导致超支 40%
4. 技术债务未记录：为赶进度引入大量临时方案，后期维护成本极高
5. 门禁标准被隐性跳过：多次"先上线后修复"，质量严重下降

**改进建议**：
1. PRD 必须包含非功能性需求，且验收标准可测试
2. 建立严格的变更控制流程：所有变更经 PM 确认，重大变更重新评审
3. 成本估算必须包含 10-15% 预留金
4. 技术债务必须实时记录，每个 Sprint 分配偿还工时
5. 门禁不可隐性跳过，所有例外必须记录

### 案例 C：资源受限模式（MVP 验证）

**项目背景**：社交小程序，2 人团队（1 全栈 + 1 设计），周期 3 周

**执行方式**：
1. 运行模式：MVP 快速验证
2. 角色合并：PM+ARCH+BE+FE+QA = 全栈开发者；UI+UX = 设计师
3. Phase 1 PRD 极简：1 页纸，聚焦"用户是否愿意为虚拟礼物付费"
4. Phase 2+3 合并：使用 Taro + 云开发，省略后端搭建
5. Phase 4+5 合并：开发者自测 + 甲方核心场景验证

**验证结果**：
- 假设成立：3 周内 200+ 用户注册，15+ 付费
- 决策：继续投入，转为标准模式，补充后端和运维

**关键经验**：
- MVP 聚焦单一假设，避免功能蔓延
- 使用云开发等成熟服务，省略基础设施搭建
- 自测 + 甲方验证可替代完整 QA 流程（MVP 阶段）

### 案例 D：冲突解决（技术方案分歧）

**项目背景**：API 网关项目，FE 和 BE 在认证方案上产生分歧

**冲突过程**：
1. FE 主张 JWT（无状态，易扩展）
2. BE 主张 Session（可控，易撤销）
3. 双方争论 2 天，无法达成一致

**解决过程**：
1. 层级 1：FE 和 BE 协商，未达成一致
2. 层级 2：PM 介入协调，提出折中方案（JWT + 黑名单），双方不完全满意
3. 层级 3：ARCH 仲裁，评估三种方案：
   - 方案 A：JWT（FE 偏好）→ 扩展性好，但撤销复杂
   - 方案 B：Session（BE 偏好）→ 可控性好，但扩展性差
   - 方案 C：JWT + Redis 黑名单（ARCH 推荐）→ 兼顾扩展性和可控性
4. 最终决策：采用方案 C，记录 ADR

**关键经验**：
- 技术冲突应在 2 天内升级，避免阻塞进度
- ARCH 仲裁时考虑项目目标而非角色偏好
- 决策必须记录 ADR，便于后续复盘

## 故障恢复流程

### 项目状态丢失恢复

```
检测到项目文档/状态损坏或丢失：
1. 立即停止所有工作，评估丢失范围
2. 尝试从版本控制恢复（Git/SVN）：
   - git show HEAD:path/to/doc.md > doc.md.backup
3. 如版本控制恢复成功：
   - 用备份替换损坏文件
   - 从备份恢复上下文
   - 标记恢复事件到项目文档
4. 如版本控制恢复失败：
   - 从目标文档和会议纪要重建最小状态
   - 标记为"重建状态，可能丢失近期历史"
   - 请求人工确认重建结果
```

### 关键人员离职应急

```
核心开发者/ARCH 离职：
1. 立即评估影响范围：
   - 该人员负责的任务清单
   - 知识文档的完整性
   - 交接文档是否存在
2. 启动知识转移：
   - 离职前 2 周内完成知识文档化
   - 代码审查记录和 ADR 回顾
   - 一对一知识转移会议
3. 调整计划：
   - 重新分配任务
   - 评估是否需要招聘替代人员或调整范围
   - 更新风险清单（进度风险：高）
4. 文档归档：
   - 将该人员的知识沉淀到知识库
   - 更新项目文档的维护责任人
```

### 甲方需求重大变更

```
甲方提出重大需求变更（影响成本/进度 > 20%）：
1. PM 记录变更请求，分析影响：
   - 成本影响：新增/减少工时
   - 进度影响：延误/提前时间
   - 范围影响：新增/删除功能
   - 风险影响：新增风险或改变现有风险等级
2. 召开变更评审会：
   - 全体角色评估影响
   - 提出接受/拒绝/调整的方案
3. 甲方确认：
   - 如接受：更新 PRD、成本估算、进度计划、风险清单
   - 如拒绝：记录拒绝理由
   - 如调整：输出调整方案，甲方确认后执行
4. 重大变更必须记录版本历史，说明变更原因和影响
```

### 安全事件应急响应

```
发现安全漏洞/数据泄露：
1. 立即响应（SEC 主导）：
   - 评估漏洞等级（CVSS）
   - 确定影响范围（数据/系统/用户）
   - 启动应急响应预案
2. 遏制（1 小时内）：
   - 隔离受影响系统
   - 阻断攻击路径
   - 保护证据（日志、快照）
3. 修复（24 小时内）：
   - 开发修复补丁
   - 测试修复方案
   - 部署修复（如紧急，可热修复后补测试）
4. 复盘（72 小时内）：
   - 根因分析
   - 影响评估（数据泄露范围、用户通知）
   - 改进措施（防止再次发生）
   - 合规报告（如适用：等保/GDPR/个保法）
```

## 合规检查清单

### 通用合规检查

```
□ 数据分类：已识别敏感数据（PII/支付/医疗）
□ 数据加密：传输加密（TLS 1.2+）+ 存储加密
□ 访问控制：最小权限原则，角色权限矩阵
□ 审计日志：关键操作有日志记录，保留 ≥ 6 个月
□ 数据备份：定期备份，恢复测试 ≥ 每季度 1 次
□ 用户授权：隐私政策、用户协议、Cookie 声明
□ 数据删除：用户数据可删除，删除后不可恢复
```

### 等保合规（中国）

```
□ 定级：已确定等保级别（1-5 级）
□ 备案：已在公安机关备案
□ 测评：已通过第三方等保测评
□ 整改：测评发现的问题已整改
□ 复测：整改后已通过复测
□ 运维：持续运维，定期复测（≥ 每年 1 次）
```

### GDPR 合规（欧盟）

```
□ 合法依据：数据处理有合法依据（同意/合同/法律义务等）
□ 同意管理：用户同意可撤回，撤回后停止处理
□ 数据主体权利：支持访问、更正、删除、可携带权
□ 数据保护影响评估（DPIA）：高风险处理已做 DPIA
□ 数据泄露通知：72 小时内通知监管机构
□ 跨境传输：数据出境有合法机制（SCC/BCR/认证）
□ DPO：如有要求，已指定数据保护官
```

### 个保法合规（中国）

```
□ 告知同意：处理个人信息前已告知并取得同意
□ 最小必要：仅收集实现目的所必要的个人信息
□ 敏感信息：处理敏感信息有单独同意
□ 跨境传输：个人信息出境已通过安全评估/认证/标准合同
□ 权利响应：支持查询、更正、删除、撤回同意
□ 数据安全：已采取必要措施保障数据安全
```

### 金融/支付合规

```
□ PCI-DSS：如涉及信用卡支付，已通过 PCI-DSS 认证
□ 支付牌照：如涉及支付业务，已取得支付牌照或与持牌机构合作
□ 反洗钱：已建立反洗钱机制（KYC/AML）
□ 资金安全：用户资金与自有资金隔离
```

### 医疗合规

```
□ HIPAA（美国）：如涉及美国患者数据，符合 HIPAA
□ 等保三级：医疗系统通常要求等保三级
□ 数据脱敏：患者数据已脱敏（非授权人员不可见原始数据）
□ 审计追踪：所有数据访问有审计日志
```

## 故障排查速查表

### 门禁失败快速排查

| 门禁失败 | 可能原因 | 排查步骤 | 解决方案 |
|---------|---------|---------|---------|
| 代码审查失败 | 逻辑错误、边界未处理 | 1. 复现问题 2. 查看审查意见 | 修复代码，重新提交 |
| 单元测试失败 | 断言错误、模拟问题 | 1. 查看失败日志 2. 分类失败类型 | 测试错误→修测试；逻辑错误→修代码 |
| 构建失败 | 依赖缺失、语法错误 | 1. 查看构建日志 2. 定位错误文件 | 修复依赖/语法，重新构建 |
| 安全测试失败 | 漏洞、配置错误 | 1. 查看漏洞报告 2. 确认影响范围 | 修复漏洞，复测 |
| 性能测试失败 | 慢查询、资源不足 | 1. 查看性能报告 2. 定位瓶颈 | 优化代码/扩容，复测 |

### 角色冲突快速排查

| 冲突场景 | 快速判断 | 解决方案 |
|---------|---------|---------|
| PM vs BA 需求理解 | 谁在做领域建模？ | PM 协调，BA 深度分析 |
| ARCH vs DEVOPS 部署 | 谁在做架构设计？ | ARCH 设计，DEVOPS 实现 |
| UI vs UX 交互 | 谁输出线框图？ | UX 输出，UI 在基础上设计 |
| QA vs SEC 安全 | 谁设计安全测试？ | SEC 设计，QA 验证修复 |
| FE vs MOBILE 多端 | 接口是否一致？ | 统一设计 Token，ARCH 仲裁 |

### 进度延误快速排查

| 延误信号 | 可能原因 | 解决方案 |
|---------|---------|---------|
| 任务持续 in-progress | 任务拆分过大 | 拆分为更小任务 |
| 阻塞频繁 | 依赖未解决 | 识别依赖，提前协调 |
| 质量反复 | 需求不明确 | PM 澄清需求，更新 PRD |
| 人员效率低 | 技术债务积累 | 分配时间偿还技术债务 |
| 范围增加 | 需求蔓延 | 严格变更控制，拒绝非必要变更 |
