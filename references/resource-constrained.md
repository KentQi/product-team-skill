# 资源受限模式与 MVP 快速验证

当项目资源不足时（一人多角、时间紧迫、预算受限），启用资源受限模式或 MVP 快速验证模式。

## 资源受限模式（Resource-Constrained Mode）

### 角色合并策略

#### 最小配置 3 人模式

```
角色 A: PM + OP（1人）
  - 总协调 + 需求理解 + 运营验收 + 项目复盘
  - 不激活 BA：直接对接甲方，使用领域模板
  - 不激活 UX：直接输出用户流程和验收标准

角色 B: ARCH + BE + DEVOPS（1人）
  - 架构设计 + 后端开发 + 部署运维
  - 不激活 SEC：基础安全审查，使用成熟安全库
  - 不激活 DATA：基础数据处理

角色 C: UI + FE + QA（1人）
  - 视觉设计 + 前端实现 + 测试验证
  - 不激活 MOBILE：FE 响应式适配
  - 不激活 UX：UI 兼任交互设计
```

#### 最小配置 2 人模式

```
角色 A: 全栈开发者（1人）
  - PM + ARCH + BE + FE + QA（简化）
  - 使用成熟框架，省略复杂架构设计
  - 测试聚焦核心流程，省略边界测试

角色 B: 设计师（1人）
  - UI + UX（如有）
  - 输出低保真原型 + 核心页面高保真
```

#### 单兵模式

```
全栈开发者（1人）：
  - 所有角色（极大简化）
  - 使用低代码/无代码平台或成熟 SaaS
  - 聚焦验证核心假设，快速交付
```

成本估算详细    → 成本估算简化（工时 × 费率）

| 门禁 | 标准模式 | 资源受限模式 | 例外记录 |
|------|---------|------------|---------|
| 代码审查 | [block] 清零 | 自测 + 走查通过 | 记录审查缺口 |
| 单元测试 | 覆盖率 ≥ 80% | 核心流程 ≥ 60% | 记录风险 |
| CI/CD | 流水线通过 | 本地构建通过 | 记录自动化缺口 |
| 安全测试 | 渗透测试通过 | 基础安全审查 | 记录安全缺口 |

| 业务验收 | 深入场景 | 核心场景验证 | 记录验证缺口 |

## MVP 快速验证模式

### 适用场景

- 产品假设未验证，需要快速验证市场/用户价值
- 时间窗口极短（2-4 周）
- 资源极少（1-3 人）
- 需要快速迭代试错

### MVP 原则

```
1. 聚焦核心假设：每个 MVP 只验证 1-2 个核心假设
2. 最小可行：只开发验证假设所需的最小功能集
3. 快速上线：2-4 周内完成开发并上线验证
4. 可测量：定义明确的成功指标（量化）
5. 可放弃：如假设不成立，可快速放弃或转向
```

### MVP 阶段合并

```
标准模式（Phase 0→5） → MVP 模式（合并为 3 个阶段）

Phase 0+1: 需求确认（1-3 天）
  - PM 确认核心假设和验证指标
  - 输出极简 PRD（1 页纸）
  - 确定最小功能集（P0 only，1-2 个 User Story）
  - 不激活任何扩展角色

Phase 2+3: 设计+开发（1-2 周）
  - 架构：使用最成熟的技术栈，零选型对比
  - 设计：低保真原型 + 核心页面高保真
  - 开发：全栈开发者或前后端各 1 人
  - 测试：开发者自测 + PM 验收
  - 部署：手动部署到测试环境

Phase 4+5: 验证+复盘（3-7 天）
  - 上线验证：收集用户反馈、验证成功指标
  - 复盘：假设验证结果（成立/不成立/需调整）
  - 决策：继续投入 / 调整方向 / 放弃
```

### MVP 门禁标准

| 门禁 | MVP 标准 | 说明 |
|------|---------|------|
| 核心假设 | 可验证 | 有明确的验证指标和测量方法 |
| 核心功能 | 可演示 | 用户能完成核心任务流程 |
| 阻塞缺陷 | 清零 | 无导致崩溃或无法使用核心功能的缺陷 |
| 安全 | 基础 | 无常见漏洞（SQL 注入、XSS），敏感数据加密 |
| 性能 | 可接受 | 核心流程响应时间 < 3 秒 |

### MVP 成功指标模板

```
假设：[我们假设...]
验证方法：[通过...验证]
成功标准：[如果...则假设成立]
指标：
  - 指标 1：[具体指标] → 目标值：[X] → 实际值：[Y]
  - 指标 2：[具体指标] → 目标值：[X] → 实际值：[Y]

验证结果：
  - 假设成立 → 进入下一迭代（标准模式或继续 MVP）
  - 假设不成立 → 分析原因，调整方向或放弃
  - 假设部分成立 → 调整假设，继续验证
```

### MVP 示例

#### 示例 A：电商小程序 MVP

```
核心假设：用户愿意为小程序内的商品支付
验证方法：上线商品展示 + 下单功能（不含支付，跳转 H5 支付）
成功标准：7 天内 100+ 用户访问，10+ 下单
最小功能集：
  - 商品列表页
  - 商品详情页
  - 购物车
  - 下单页（跳转 H5 支付）
省略功能：
  - 用户系统（用微信 openid）
  - 订单管理（后台手动处理）
  - 优惠券
  - 推荐算法
  - 物流跟踪
团队配置：1 全栈 + 1 UI（2 周）
```

#### 示例 B：企业协作工具 MVP

```
核心假设：团队愿意为实时协作功能付费
验证方法：上线文档实时协作（仅支持文本）
成功标准：10 个团队试用，3 个团队愿意付费
最小功能集：
  - 文档创建和编辑
  - 实时协作（WebSocket）
  - 基础权限（只读/编辑）
省略功能：
  - 评论和 @ 提及
  - 版本历史
  - 导出 PDF/Word
  - 移动端适配
  - SSO 集成
团队配置：1 全栈 + 1 设计（3 周）
```

## 优先级调整策略

资源受限时，按以下优先级调整：

```
1. P0 功能：必须完整交付（核心用户价值）
   - 无妥协，无例外
   - 所有资源优先保障 P0

2. P1 功能：可简化实现（核心但非阻塞）
   - 功能可简化，但可用
   - 例如：搜索功能，标准版支持全文检索，简化版支持关键词匹配

3. P2 功能：可推迟到后续迭代（锦上添花）
   - 当前迭代完全不开发
   - 记录到后续迭代计划

4. 非功能性需求：设最低门槛，后续优化
   - 性能：核心流程响应时间 < 3 秒（标准版 < 1 秒）
   - 安全：基础防护（标准版完整安全方案）
   - 兼容性：最新 2 个浏览器版本（标准版最新 3 个 + IE11）
   - 可用性：核心流程可完成（标准版 WCAG 2.1 AA）
```

## 资源受限模式启动检查单

```
□ 已确认资源受限（人力/时间/预算）
□ 已选择运行模式：资源受限 / MVP
□ 已确定角色合并方案（3人/2人/单兵）
□ 已确认 P0 功能清单（聚焦核心）
□ 已调整质量门禁标准（记录例外）
□ 已确认风险接受声明（甲方/管理层）
□ 已制定后续迭代补齐计划（资源恢复后）
```
